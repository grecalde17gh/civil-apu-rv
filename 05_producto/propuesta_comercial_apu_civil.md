# Propuesta Comercial — Sistema de Presupuestos y Análisis de Precios Unitarios

## 1. Propósito del documento

Este documento define una propuesta comercial preliminar para presentar el sistema de presupuestos y análisis de precios unitarios a posibles usuarios, clientes piloto o aliados estratégicos.

El objetivo es traducir el producto técnico en una oferta clara, entendible y atractiva para el mercado.

---

## 3. Descripción corta del producto

```text
APU Civil Pro Ecuador es una aplicación web para crear presupuestos de obra y análisis de precios unitarios de forma ordenada, transparente y validada técnicamente, reemplazando procesos manuales dispersos en Excel por una base de datos estructurada y cálculos confiables.
```

---

## 4. Problema que resuelve

Muchos ingenieros civiles elaboran presupuestos y análisis de precios unitarios en archivos Excel manuales.

Aunque Excel es flexible, con el tiempo genera problemas como:

1. Fórmulas rotas.
2. Referencias inválidas.
3. Duplicación de datos.
4. Dificultad para actualizar precios.
5. Rubros difíciles de reutilizar.
6. Falta de trazabilidad.
7. Riesgo de errores en presupuestos.
8. Exportaciones poco estandarizadas.
9. Dependencia de archivos personales.
10. Dificultad para escalar a equipos de trabajo.

---

## 5. Solución propuesta

El producto propone una aplicación web que permite:

1. Administrar materiales.
2. Administrar mano de obra.
3. Administrar equipos y herramientas.
4. Crear rubros de construcción.
5. Elaborar análisis de precios unitarios.
6. Calcular precios unitarios automáticamente.
7. Crear presupuestos de obra.
8. Reutilizar rubros.
9. Exportar presupuestos a Excel y PDF.
10. Validar técnicamente rubros y cálculos.

---

## 6. Propuesta de valor

Reducir el tiempo y los errores en la elaboración de presupuestos y APUs, manteniendo trazabilidad, orden y validación técnica.

Esto evita errores silenciosos.

---

Los rubros pueden reutilizarse, editarse, validarse y duplicarse.

---

## 9.6. Preparado para IA futura

La IA no se usará como adorno, sino como futura asistencia para:

- Sugerir rubros.
- Detectar inconsistencias.
- Comparar presupuestos.
- Generar observaciones técnicas.

## 11. Lo que no debe prometerse todavía

En la etapa inicial no se debe prometer:

1. Que reemplaza todos los sistemas existentes.
2. Que cumple automáticamente con todos los formatos de contratación pública.
3. Que la IA crea presupuestos perfectos.
4. Que los precios se actualizan solos desde proveedores.
5. Que sirve para todo tipo de obra sin configuración.
6. Que no requiere validación técnica.
7. Que está listo como SaaS masivo desde el día uno.

Prometer menos y cumplir más. Es menos glamuroso, pero mucho más rentable.

## 18. Objeciones esperadas y respuestas

## 18.1. “Ya uso Excel”

Respuesta:

```text
Perfecto. La idea no es negar Excel, sino reducir los riesgos de manejar toda la lógica en archivos manuales. El sistema puede incluso exportar a Excel, pero mantiene los datos y cálculos principales más ordenados y trazables.
```

---

## 18.2. “Mi Excel ya está personalizado”

Respuesta:

```text
Justamente por eso el primer paso puede ser revisar tu Excel y entender qué parte vale la pena mantener, qué parte se puede estructurar mejor y qué errores o riesgos existen.
```

---

## 18.3. “No quiero depender de internet”

Respuesta:

```text
La primera versión puede trabajarse localmente o en un entorno controlado. Más adelante se puede evaluar una versión web según la necesidad del cliente.
```

---

## 18.4. “¿La IA va a inventar precios?”

Respuesta:

```text
No. La IA no debe inventar precios ni aprobar rubros. Primero se construye un motor técnico confiable. La IA, si se integra, será solo una asistencia para sugerir, revisar o comparar, siempre con validación humana.
```

---

## 18.5. “¿Sirve para obra pública?”

Respuesta:

```text
Puede adaptarse a formatos y necesidades de obra pública, pero cada formato específico debe validarse. En la primera fase nos enfocamos en que el cálculo APU y el presupuesto sean técnicamente correctos.
```

## 24. Riesgos comerciales

| Riesgo | Impacto | Mitigación |
|---|---|---|
| Cliente piensa que es solo otro Excel | Medio | Mostrar base estructurada, snapshots y validación. |
| Cliente espera IA mágica | Alto | Posicionar IA como asistencia futura. |
| Mercado no quiere pagar software | Medio | Ofrecer implementación acompañada. |
| Producto aún inmaduro | Alto | Iniciar con pilotos controlados. |
| Soporte consume demasiado tiempo | Alto | Limitar alcance del piloto. |
| Cada cliente quiere personalización distinta | Alto | Definir núcleo común y parametrizar solo lo necesario. |
